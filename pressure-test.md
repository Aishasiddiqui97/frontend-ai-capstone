# Pressure Test — Portfolio Sitemap Review

**Reviewer:** Claude (AI Tutor)
**Document under review:** `portfolio-sitemap.md`
**Author:** Aisha Siddiqui

---

## 1. Does every page support my one action?

The primary action is: *Contact me for an internship or junior frontend developer opportunity.*

| Page | Supports the action? | How? |
|---|---|---|
| Home | Yes | Establishes identity and includes a direct CTA button directing visitors to contact |
| Projects | Yes | Provides evidence that I can do the work — builds trust before the ask |
| About | Indirectly | Humanises me and shows trajectory, but does not directly push toward contact |
| Contact | Yes | This page *is* the action — the destination of every CTA |

**Verdict:** Largely yes. The About page is the weakest link — it supports the case for hiring me but does not actively drive toward the contact action. Consider adding a soft CTA at the bottom of the About page (e.g., "Interested in working together? Let's talk.").

---

## 2. Does every page strengthen my proof statement?

> *I build clean, modern, AI-assisted frontend applications using React, Next.js and TypeScript while continuously improving through real-world projects.*

**Home** — Reinforces the statement directly in the headline and introduction. Strong.

**Projects** — This is where the statement is proven. Each project must explicitly show the tech stack (React, Next.js, TypeScript, Tailwind CSS) and mention how AI assisted the process. If projects don't include AI-assisted development notes, the proof statement feels unsupported.

**About** — Supports the "continuously improving" part of the statement by showing my learning journey. Weak on the "build clean, modern" aspect — expected since this is a personal page.

**Contact** — Does not directly strengthen the statement, but it enables the outcome the statement is meant to drive. That is acceptable.

**Verdict:** Strong overall, but the Projects page must carry the heaviest load. If projects lack AI-assistance notes, the proof statement reads as hollow.

---

## 3. What is unnecessary?

Nothing in the sitemap is truly unnecessary. The exclusion of a blog, testimonial section, and skills page are well-reasoned. The sitemap is deliberately minimal, and every remaining page has a justified purpose.

**However:** A "Resume" link is mentioned only in passing under the Home page content. It deserves more visibility. If a downloadable PDF resume exists, it should be called out as its own element, not buried in a content list.

---

## 4. What is missing?

| Missing Element | Why It Matters |
|---|---|
| Visible skills / tech stack section | Recruiters scan for tech stack immediately. A skills summary on the Home page (badges or a short list) saves them effort. |
| Downloadable PDF resume (prominent) | Many recruiters expect a one-page PDF they can save or print. It should be a top-level callout, not a footnote. |
| Project impact metrics | "Built a weather app" is weak. "Built a weather app that serves 500 daily users with 95+ Lighthouse score" is strong. |
| GitHub activity indicator | A contribution graph or recent commit count on the Home page signals active, consistent work. |
| Soft CTA on every page | Not every page has a call-to-action. The Home and Contact pages do; Projects and About rely on users navigating further. Every page should offer a next step. |

---

## 5. How would a recruiter experience it?

A recruiter lands on Home. Within seconds they see my name, title, tech stack, and a clear CTA. Good. They browse to Projects — clean cards with tech tags, live demos, and repos. Good. They check About to see if I'm self-taught or formally trained, and whether my learning trajectory matches their needs. The page covers it. Finally they visit Contact and send a message.

**The flow works.** The risk is that a recruiter in a hurry skips Projects and About entirely, looks only at Home, and wants a resume immediately. If the resume link is not prominent, they may bounce.

## 7. Five Improvements

1. **Add a visible tech stack banner to the Home page** — A row of coloured badges (React, Next.js, TypeScript, Tailwind, etc.) so recruiters confirm skill fit in under 2 seconds.

2. **Surface the downloadable resume as a top-level element** — A "Download Resume" button in the navigation bar or hero section, not buried in a paragraph.

3. **Include AI-assistance notes in every project write-up** — The proof statement promises AI-assisted development. Each project must briefly state how AI contributed (e.g., "ChatGPT helped draft the initial component structure; I refactored for performance and accessibility.").

4. **Add a soft CTA on every page** — The bottom of Projects and About should include a line like "See something you like? Reach out." This keeps the primary action visible throughout the browsing experience.

5. **Add one measurable outcome per project** — Replace generic descriptions with impact metrics where possible. "Built a task manager" would become "Built a task manager with 100% test coverage and <200ms API response times."
