# Portfolio Sitemap — Aisha Siddiqui

**Target:** Recruiters, hiring managers, startup founders, internship reviewers
**Primary action:** Contact me for an internship or junior frontend developer opportunity
**Proof statement:** *I build clean, modern, AI-assisted frontend applications using React, Next.js and TypeScript while continuously improving through real-world projects.*

---

## Home

**Purpose:** Make an immediate, professional impression. Establish who I am, what I do, and what I want — within the first 5 seconds.

**Content:**
- Full name and title (Frontend Developer / AI Engineering Learner)
- One-line headline reinforcing the proof statement
- Short introduction (2–3 sentences)
- Featured projects preview with thumbnails and tech tags
- Prominent call-to-action button: "Looking for an intern? Let's talk."
- Links to GitHub, LinkedIn, and downloadable resume

**Why it earns its place:** The homepage is the front door. Recruiters decide within seconds whether to stay or leave. Every element here must justify continued browsing and drive toward the contact action.

---

## Projects / Case Studies

**Purpose:** Provide concrete evidence that I can build real applications. This is where the proof statement is validated.

**Content:**
- 3–4 featured projects (FlyRank assignments, portfolio apps, open-source contributions)
- Each project includes:
  - Title, one-line summary, live demo link, and GitHub repository link
  - Tech stack badges (React, Next.js, TypeScript, Tailwind CSS, etc.)
  - Key challenge solved and what I learned
  - AI-assisted development highlights (e.g., "Used Claude to generate component scaffolding, reviewed and refined manually")

**Why it earns its place:** A portfolio without projects is just words. This section provides the evidence recruiters need to evaluate my skill level, code quality, and problem-solving ability. It directly supports the proof statement by showing *what* I build.

---

## About

**Purpose:** Humanise the portfolio. Show the person behind the code — my journey, motivation, and trajectory.

**Content:**
- Brief background (career shift or learning path into tech)
- Current focus areas (Next.js, TypeScript, AI Engineering)
- Learning philosophy ("I believe in building real things to learn real skills.")
- Career goal (remote frontend developer or AI engineering internship)
- A photo (optional, but recommended for connection)

**Why it earns its place:** Recruiters hire people, not skill lists. The About page answers the unspoken question: "Who is this person and would I want to work with them?" It complements the proof statement by showing the *continuous improvement* part of my claim.

---

## Contact

**Purpose:** Remove all friction from the primary action. Make it trivially easy to reach out.

**Content:**
- Simple contact form (name, email, message)
- Direct email address
- LinkedIn and GitHub profile links
- Clear message: "I'm actively looking for internship and junior developer opportunities. Let's talk."

**Why it earns its place:** This is the conversion page — the entire portfolio exists to drive visitors here. Without it, the primary action has no destination. Every other page leads to this one.

---

## Why Additional Pages Are Unnecessary

| Page | Reason Excluded |
|---|---|
| Blog | Adds maintenance overhead without directly supporting the internship goal. Recruiters rarely evaluate blog content for junior roles. |
| Resume page | A downloadable PDF resume linked from Home and About is more practical. A full page duplicates About content. |
| Testimonials | As a junior, gathering meaningful testimonials is premature. A single mentor quote on About is sufficient if available. |
| Skills page | Skills are already present on Home (headline, tags) and within each project card. A separate page is redundant. |
| Gallery / Visual work | Not applicable for a frontend developer portfolio focused on applications rather than design. |

**Principle:** Every page must either strengthen the proof statement or directly enable the primary action. Pages that do neither are distractions.
